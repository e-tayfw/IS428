# MC1

## Project Overview

MC1 is a Quarto-based visual analytics website focused on exploring artist collaboration networks from JSON data. The current analysis centers on **Sailor Shift**, with the goal of understanding how her relationships with featured artists, producers, and collaborators shape her creative reach and influence within the broader music industry.

Rather than treating collaboration as a simple list of names, this project is intended to study the structure behind those connections. By examining recurring partnerships and network patterns, the site can be used to show who Sailor Shift works with most often, how central those collaborators are, and what that reveals about her impact in the industry.

## Project Goals

- Analyze JSON-based music or artist relationship data
- Investigate Sailor Shift's collaboration network
- Identify important collaborators and repeated partnerships
- Present the analysis through a clean multi-page Quarto website
- Provide a structure that can later hold charts, network diagrams, and a final poster

## Project Stack

- **Quarto** for site generation
- **QMD (`.qmd`) files** for page content
- **CSS** for custom styling
- **HTML output** generated by Quarto into the `_site/` folder

## Directory and File Description

### Core Project Files

- **`_quarto.yml`**
  Main Quarto configuration file. This controls the website type, output folder, site title, and the top navigation bar.

- **`styles.css`**
  Shared stylesheet for the site. Use this file to change colors, layout, spacing, buttons, cards, typography, and general visual styling.

- **`README.md`**
  Project documentation. This file explains the purpose of the project, how to run it, and what each major file is for.

### QMD Page Files

- **`index.qmd`**
  The homepage of the website. This acts as the landing page and introduces the project. It links users to the Proposal, Methodology, and Poster pages through the navigation bar and homepage buttons.

- **`proposal.qmd`**
  The proposal page. This page is meant for the project rationale and planning content, such as:
  - problem statement
  - objectives
  - scope
  - expected outcomes

- **`methodology.qmd`**
  The methodology page. This page should explain how the analysis is conducted, including:
  - data sources
  - JSON structure or schema
  - data cleaning and preparation
  - analytical approach
  - tools used

- **`poster.qmd`**
  The poster page. This page is intended for the final presentation output, such as:
  - poster summary
  - final charts or visuals
  - main findings
  - project conclusion

### Generated and Local-Only Folders

- **`_site/`**
  This is the rendered website output folder generated by Quarto. It contains the final HTML files and supporting assets used by the website. You should **not edit files here manually**, because Quarto will regenerate them.

- **`.quarto/`**
  This is Quarto's local working directory for cache and temporary files. It is created automatically and should not be committed.

## Website Navigation

The website uses a top navigation bar configured in `_quarto.yml`.

Current navigation:

- Home
- Proposal
- Methodology
- Poster

Each navigation item links to a separate `.qmd` file, which Quarto renders into its own HTML page. That means:

- `index.qmd` becomes `index.html`
- `proposal.qmd` becomes `proposal.html`
- `methodology.qmd` becomes `methodology.html`
- `poster.qmd` becomes `poster.html`

This setup ensures that clicking `Proposal` or `Methodology` opens a proper standalone page instead of only jumping to a section on the homepage.

## How to Start the Project

### Prerequisite

Make sure Quarto is installed on your machine.

Check with:

```bash
quarto --version
```

If Quarto is installed correctly, it will print a version number.

## Running the Project Locally

### Option 1: Start a live preview server

Use this during development:

```bash
quarto preview
```

What this does:

- renders the website
- starts a local development server
- watches for file changes
- refreshes the site when you edit `.qmd` or `.css` files

How to use it:

1. Open the terminal in this project folder
2. Run `quarto preview`
3. Open the local URL shown in the terminal
4. Edit your files and save them
5. Refresh is usually automatic
6. Stop the server with `Ctrl + C`

If you want to use a fixed port and prevent Quarto from opening a browser automatically:

```bash
quarto preview --no-browser --port 4200
```

### Option 2: Render the full site output

Use this when you want to generate the static website files:

```bash
quarto render
```

What this does:

- builds the site into the `_site/` folder
- creates the HTML files for each page
- prepares the site for sharing or deployment

Use `quarto render` when:

- you want a final build
- you want to verify the site compiles without errors
- you are preparing the site for submission or hosting

## Recommended Workflow

1. Edit content in the `.qmd` files
2. Adjust styling in `styles.css`
3. Run `quarto preview` while working
4. Run `quarto render` before final submission
5. Commit source files only, not generated output

## Styling Notes

The site styling is centralized in `styles.css`.

This means:

- page layout can be changed without rewriting all page content
- the homepage and content pages can share the same visual design
- colors, cards, buttons, spacing, and typography can be updated in one place

If you want to redesign the site later, `styles.css` is the main file to update.

## Working With JSON Data

This project is intended for JSON-based analysis of artist relationships. A good future workflow would be:

1. Store raw or working JSON files in a local data folder such as `data/` or `datasets/`
2. Keep large or temporary datasets out of Git
3. Summarize the dataset and analysis process on `methodology.qmd`
4. Present the findings on `poster.qmd` and possibly the homepage

If your JSON files are large, sensitive, or frequently changing, it is better to keep them ignored through `.gitignore`.

## Git Ignore Notes

This project uses `.gitignore` to prevent local-only and generated files from being committed.

Important ignored paths include:

- `.quarto/` for local Quarto cache files
- `_site/` for rendered website output
- `**/*.quarto_ipynb` for temporary notebook-related files
- `data/` and `datasets/` contents for local datasets

In general:

- commit source files such as `.qmd`, `.css`, `_quarto.yml`, and `README.md`
- do not commit generated HTML output unless your workflow specifically requires it
- do not commit raw local datasets unless the team agrees they should be version-controlled
